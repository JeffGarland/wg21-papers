\subsection{LEWG at Kona 2017}
\begin{itemize}
  \item \wgUnanimous{Want \code{operator<<(signed)} to work except where it's undefined for the underlying integer?}

  \item \wgPoll{Should there be overloads for \code{where} and (\mask and \datapar) reductions with \bool/builtin types in place of \mask/\datapar?}{1&5&6&1&0}

  \item \wgUnanimous{Should there be a named “widen” function that widens the element type \type T from (e.g.) \intt to \type{long}, but rejects \intt to \short?
  The number of elements is not changed.}
  cf. \autoref{sec:widen}

\item \wgUnanimous{Should there be a named “concat” functions that concatenates several \datapar{}s with the same element type, but potentially different length?\\
  The latter two operations are currently lumped together in \code{datapar_cast}.\\
  The widen / concat questions, I think, comes down to whether we want the current \code{datapar_cast} or not.}
  cf. \autoref{sec:split and concat}

\item \wgUnanimous{We still have a few open ends on implicit conversions (\url{https://github.com/mattkretz/wg21-papers/issues/26}, \url{https://github.com/mattkretz/wg21-papers/issues/3}).
  The current paper is rather strict, there may be room for more implicit conversions without introducing safety problems.
Should we postpone any changes in this area to after TS feedback?}

\item \noindent Poll: Keep or drop \code{datapar_abi::max_fixed_size}? (cf. \url{https://github.com/mattkretz/wg21-papers/issues/38})\\
  \hspace*{2em} Authors will come back with a proposal\par\medskip
  cf. \autoref{sec:maxfixedsize}

\item \wgUnanimous{Add a list of possible names for “datapar” and “where” to the paper.}
  cf. \autoref{sec:Naming}
\end{itemize}
